1 · The system in one view

what computes what, and where

datagen
seeded market sim→ Parquet
columnar files→ DuckDB
star schema, 10.4M rows→ engine
Kotlin: TWR · Brinson · Cariño · risk→ data.json
baked per portfolio→ dashboard
static page, no server

A Kotlin pipeline generates a synthetic two-year equity market, lands it in DuckDB (an embedded columnar OLAP database) as a star schema with a 10,377,750-row daily positions fact table, computes performance analytics with a tested engine, and bakes the results into a 1 MB data.json. The dashboard is a static page: no backend, no live queries — GitHub Pages serves three files and the browser renders them with Chart.js.

The headline: the core attribution query aggregates the 10.4M-row fact table in 1.31 s inside DuckDB — 6.7× faster than pulling the rows into the JVM and aggregating there, with both implementations agreeing to within 5.2×10−17. The entire speedup is where the math runs: the optimized path moves 550 result rows across the JDBC boundary instead of 10.4 million input rows.

Everything here is reproducible: a fixed random seed (42), golden tests pinning hand-worked numbers to 1e-12, and invariant tests enforcing the accounting identities to 1e-10.

synthetic, but built to behave like the real thing

How the market is simulated

Prices follow geometric Brownian motion with three shock layers — a market-wide shock, a sector shock (so sectors move together, which is what makes sector attribution meaningful), and an idiosyncratic shock per stock. About 60% of securities pay quarterly dividends, modeled as immediately reinvested at the close. Portfolios receive periodic external cash flows (invested pro-rata at the prior close) and make occasional value-neutral rebalance trades. Each portfolio persistently overweights three “favorite” sectors — that tilt is what produces nonzero allocation effects.

What's baked into the dashboard

data.json carries, per portfolio: the daily return series (and one shared benchmark series), the full-period Cariño-linked attribution by sector, top/bottom contributors, and weekly sector weights. Daily series are what let the range buttons recompute returns and risk client-side; attribution stays full-period because recomputing it for arbitrary ranges would need the per-day×sector effect matrix (~30 MB). Against the live backend the API is shaped by the page's access pattern — /api/portfolios and /api/market at boot, /api/portfolio/{id} on selection, and the heavy weights matrix on its own endpoint after first paint — so the first render costs ~6.5 KB over the wire (gzip) instead of the 1 MB document.

3 · Time-Weighted Return (TWR)

the return of the strategy, not the client's deposit timing

A portfolio that receives a big deposit right before a bad week looks terrible on a naive start/end calculation — even if every investment decision was sound. TWR removes that distortion: cut time into sub-periods at each external cash flow, compute each sub-period's return in isolation, and compound them. Cash flows are assumed to arrive at the beginning of the day (the w = 1 convention), so the flow is in the denominator — it had the whole day to earn:

r_t = (MV_end − MV_begin − CF) / (MV_begin + CF)        daily sub-period return
TWR = ∏(1 + r_t) − 1                                     geometric linking
annualized = (1 + TWR)^(365.25 / calendar days) − 1      only for periods ≥ 1 year

Worked example (the repo's golden test)

Four securities, three days, one +1,020 cash flow at the start of day 2:

day 1:  (4,040 − 4,000 − 0)     / 4,000           = +1.0000%
day 2:  (5,150 − 4,040 − 1,020) / (4,040 + 1,020) = 90/5,060 = +1.7787%
day 3:  (5,185 − 5,150)         / 5,150           = +0.6796%
TWR  =  1.01 × 1.017787 × 1.006796 − 1 = 3537/101200 = +3.4951%

Note day 2: the naive calc (5,150/4,040 − 1 = +27%) would credit the deposit as “return.” TWR strips it out. Dividends are not external flows — they're income the portfolio earned, so they stay in the numerator (each security's daily return is a total return, (p_t + dividend)/p_(t−1) − 1).

4 · Brinson-Fachler attribution

decomposing where active return came from

Active return = portfolio return − benchmark return. Attribution splits it by sector into three effects, using four inputs per sector i: portfolio weight wp, benchmark weight wb, portfolio sector return rp, benchmark sector return rb_i (and the total benchmark return rb):

Allocation   A = (wp − wb) × (rb_i − rb)   did over/under-weighting the sector help?
Selection    S = wb × (rp − rb_i)          within the sector, did you pick better stocks?
Interaction  I = (wp − wb) × (rp − rb_i)   the cross-term: extra weight × extra skill

Intuition: allocation rewards overweighting sectors that beat the overall benchmark (note the Brinson-Fachler refinement — it compares the sector to rb, so overweighting a positive-but-below-average sector correctly scores negative). Selection rewards beating the sector's own benchmark return with your picks, sized by the benchmark weight. Interaction is what's left so the identity closes:

Σ (A + S + I) = active return        exactly — enforced in tests to 1e-10

Worked example (golden test, day 1)

Two sectors. Portfolio holds 50/50; benchmark holds Tech 60 / Energy 40. Tech returns 0% in the portfolio but +2.5% in the benchmark; Energy +2% in both. Benchmark total rb = 0.6×2.5% + 0.4×2% = 2.3%; portfolio = 1.0%; active = −1.3%.

SectorASI
Tech  (underweight a strong sector, bad picks) −0.0002−0.0150+0.0025
Energy (overweight a below-average sector) −0.000300
Sum−0.0130 = active ✓

Weights are prior-close market-value weights; the engine defines an unheld sector's rp := rb_i so it shows pure allocation and no fictitious selection. Per-security contribution is the simpler cousin: c = w × r, summing exactly to the portfolio return — that's the Contributors panel.

5 · Cariño linking

why daily effects can't just be added — and the fix

The Brinson identity holds per day. But returns compound geometrically while effects add arithmetically, so summed daily effects drift away from the multi-period active return (in this data, by ~180 bps over two years). Cariño log-linking rescales each day's effects so they reconcile exactly:

k(a,b) = (ln(1+a) − ln(1+b)) / (a − b)         limit 1/(1+a) as b → a
k_t = k(rp_t, rb_t)      per-day        k = k(R_p, R_b)      full-period
scaled effect = effect × k_t / k
⇒  Σ_t Σ_i scaled effects = R_p − R_b        the geometric active return, exactly

Worked example

periods: (rp, rb) = (10%, 5%), then (−2%, +1%)     R_p = 7.80%, R_b = 6.05%
k₁ = 0.930400312698     k₂ = 1.005101272356     k = 0.935255855097
factor₁ = 0.994808327183  → scaled active +4.9740%
factor₂ = 1.074680545305  → scaled active −3.2240%
                                     sum = +1.7500% = R_p − R_b ✓
Interview nuance: the identity constrains only the sum. Cariño, Menchero, and GRAP distribute the reconciliation differently — per-sector linked numbers are convention-dependent even though every method hits the same total. This engine implements Cariño and says so in its methodology.

6 · Risk metrics

the KPI cards' other half

Tracking error  TE = stdev(daily active returns) × √252      annualized; "how far do I stray?"
Info ratio      IR = mean(daily active) × 252 / TE           active return per unit of risk taken
Max drawdown    worst peak-to-trough fall of the compounded portfolio path

Rules of thumb: TE of 2–6% is a typical active equity mandate; IR above ~0.5 is good and above 1.0 is excellent over meaningful horizons. Max drawdown answers the client's most visceral question — “how much could I have been down?” IR on short ranges (the 3M button) is statistically noisy; treat it as indicative, not significant.

8 · Walkthrough: Portfolio 007, full period

the narrative an interviewer should hear

“Over 504 trading days the portfolio returned +33.18% against the benchmark's +25.51% — +7.67% active. It earned that with a tracking error of just 2.49%, so the information ratio is 1.19: strong risk-adjusted active management, not a lucky concentrated bet. The worst peak-to-trough fall was −16.97%.”

“Where did the 767 basis points come from? The waterfall says Real Estate dominates: +467.9 bps, almost all selection (+212.5) and interaction (+273.5) — the portfolio tripled the benchmark's weight (15.2% vs 6.5%) and picked the right names, so the cross-term is large and positive. Energy added +266.6 bps the opposite way: mostly allocation (+213.1) from holding half the benchmark's weight (5.2% vs 10.5%) in a sector that lagged. The big detractor is Consumer Staples, −204.7 bps, driven by selection (−188.5) — right-ish weight, wrong stocks.”

“The sector totals are Cariño-linked, so they sum to +767.2 bps — exactly the geometric active return on the cards. That reconciliation is enforced by tests, not approximate.”

Practice this paragraph until it's fluid — it demonstrates, in one breath, that you understand performance, risk, attribution, and the linking subtlety.

9 · Interview Q&A

the questions to expect, with honest answers

Why TWR rather than a simple or money-weighted return?
Simple start/end return is distorted by external cash flows the manager doesn't control. TWR isolates strategy skill by compounding flow-free sub-periods. Money-weighted return (IRR) answers the complementary question — the client's experience including their timing — and is on the roadmap; the two diverge exactly when large flows precede strong moves.
Why beginning-of-day flows (w=1)? What about Modified Dietz?
A convention had to be picked and enforced; w=1 says the flow was invested all day, which matches the generator's mechanics (flows buy at the prior close). Modified Dietz (w=0.5) approximates mid-period flows when timestamps are unknown. The convention is documented, golden-tested, and isolated in one function — swapping it is a one-line change.
Why report interaction separately? Many shops fold it into selection.
Transparency first: folding hides that the cross-term can dominate (see Real Estate, where interaction exceeds selection). Two-effect reporting is a presentation choice on top of the same math, and trivially derivable from this output.
Why do attribution totals match the active return exactly?
Per day it's an algebraic identity of Brinson-Fachler. Across days it's Cariño log-linking — without it, summed daily effects drift from the geometric active return. Both are enforced as test invariants (1e-10), not eyeballed.
Where does the 6.7× query speedup come from?
Not from better math — both paths compute identical numbers (verified to ~1e-17). The naive path serializes 10.4M rows across JDBC and aggregates row-at-a-time on the JVM heap; the optimized path runs one set-based SQL statement — a window-function lag for prior-close weights, hash joins, group-bys — inside DuckDB's vectorized columnar engine and returns 550 rows. Move results, not rows. The annotated EXPLAIN ANALYZE is in the repo.
Why is the dashboard static? Is that a real architecture?
Performance analytics are read-heavy over end-of-day batch data — the same reasoning behind materialized views and OLAP cubes. All computation happens at build time; serving is three static files on a CDN. The trade-offs are owned: data is stale until re-baked, sub-range attribution isn't available, and nothing here is private. A live server would be a thin JSON layer over the same engine functions.
What are the known limitations?
Single currency (USD); daily gapless position snapshots assumed (real custodian data violates this); the benchmark must span every held sector; per-sector linked numbers are Cariño-convention-specific; IR is noisy on short ranges; the benchmark aggregation paths use arithmetic sums by design (linking is a report-layer concern). Each is documented rather than hidden.
How do you know the numbers are right?
Three layers: golden tests reproduce hand-worked examples (documented in METHODOLOGY.md before the engine was written) to 1e-12, through both pure Kotlin and the real SQL; invariant tests enforce the accounting identities on randomized inputs; equivalence tests require the naive and optimized implementations to agree. The dashboard's client-side math mirrors documented formulas and the baked full-period values are computed by the tested engine.
